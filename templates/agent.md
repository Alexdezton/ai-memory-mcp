# AGENT.md — AI MEMORY PROTOCOL

**ЭТО ГЛАВНЫЙ ФАЙЛ.** При LOADER агент читает `core_rules.md` + `tasks.md`. Этот файл — карта AI_MEMORY.

**⚠️ ЛИМИТ: agent.md не должен превышать 600 строк.** При приближении к 500 строкам — вынести подробности в `specifications/` или `other/`. (Подробное правило: core_rules.md — File Size Control Protocol)

## 1. Роль агента

Агент работает с проектом через AI_MEMORY. Он:
- Управляет контекстом проекта через AI_MEMORY
- Планирует и делегирует задачи субагентам (implementation, research, review, test)
- Контролирует качество и соблюдение протоколов
- Обновляет документацию проекта

## 2. Протокол работы с AI_MEMORY

### 2.1 Старт сессии (LOADER) — Минимальный контекст
```
Команда пользователя: LOADER [TIME]
```

**Что делает оркестратор:**
1. Читает ТОЛЬКО 3 файла:
   - `core_rules.md` → Правила, PROJECT_STATE, prepend
   - `tasks.md` → Задачи, ACTIVE_TASK, NEXT_STEP, BLOCKERS
   - `checkpoint-latest.json` → Состояние (если есть)
2. Обновляет `LAST_LOADER.json` с timestamp
3. Выводит: "AI_MEMORY LOADED" + ACTIVE_TASK, NEXT_STEP, BLOCKERS, PROJECT_STATE
4. Начинает выполнять NEXT_STEP

**Остальные файлы читаются ПО МЕРЕ НЕОБХОДИМОСТИ:**
| Нужно | Читает |
|-------|--------|
| Понять архитектуру | `architecture_map.md` |
| Бизнес-логика | `specifications/[feature].md` |
| Прошлые решения | `decisions.md` |
| История сессий | `session_log.md` |
| Процедуры | `procedures.md` |
| О проекте | `project_overview.md` |
| Карта знаний | `knowledge_index.md` |

### 2.2 Карта всех файлов AI_MEMORY

**Главное правило:** Агент читает `core_rules.md` + `tasks.md` при старте. Остальное — по необходимости через эту карту.

```

    template.md              ← Шаблон спецификации
    [feature].md             ← Конкретная фича

```

### 2.5 Завершение сессии (EOS)
```
Команда пользователя: EOS
```

**Что делает оркестратор:**
1. Читает `AI_MEMORY/EOS.md` и следует протоколу
2. **PHASE 1: Information Gathering** — собирает ВСЕ изменения
3. **PHASE 2: Summary & Update** — создаёт 5-точечный summary, обновляет файлы
4. Создаёт `checkpoint-latest.json`
5. Выводит: "CHECKPOINT SAVED: {ACTIVE_TASK, NEXT_STEP, BLOCKERS}"

## 3. Управление субагентами

### 3.1 Запуск субагента

Оркестратор запускает субагентов через `new_task` с указанием режима:

| Режим | Слог | Когда использовать |
|-------|------|-------------------|
| Architect | `architect` | Планирование, дизайн архитектуры, review решений |
| Code | `code` | Написание кода, создание/редактирование файлов |
| Ask | `ask` | Объяснения, документация, анализ существующего кода |
| Debug | `debug` | Поиск багов, анализ ошибок, исправление проблем |

### 3.2 Процесс делегирования

```yaml
1. Анализ задачи:
   - Прочитать задачу из tasks.md
   - Определить тип субагента
   - Подготовить контекст (файлы, спеки, архитектуру)

2. Запуск субагента:
   - new_task(mode: [режим], message: [описание задачи], todos: [список шагов])
   - Дождаться завершения субагента

3. Контроль результата:
   - Прочитать результат субагента
   - Проверить корректность выполнения
   - При ошибке — анализ, корректировка, перезапуск

4. Интеграция:
   - Объединить результат в основной контекст
   - Обновить документацию
   - Сообщить пользователю о прогрессе
```

### 3.3 Правила запуска субагентов

- **Всегда** передавай полный контекст задачи (файлы, спеки, решения)
- **Всегда** указывай `todos` — список шагов для выполнения
- **Всегда** читай результат субагента перед следующим шагом
- **Никогда** не запускай несколько субагентов одновременно для зависимых задач
- **Можно** запускать параллельно независимые задачи

## 4.1 📝 PREPEND SYSTEM — Экономия токенов

**ПРАВИЛО:** Prepend-система применяется ТОЛЬКО к файлам с пометкой `⚠️ PREPEND FILE`. НЕ применяй prepend ко всем файлам — только к тем, где явно указано. Остальные используют append или свой формат.

**Зачем:** Агент видит свежие записи сразу при чтении первых строк, не тратит токены на прочтение всей истории.

**Файлы с prepend (новые записи ВНАЧАЛЕ):**
- `decisions.md` — решения проекта
- `session_log.md` — логи сессий

**Файлы с естественной логикой (append):**
- `tasks.md` — новые задачи в конец, выполненные остаются внизу как история
- `brainstorm/` — каждая тема = отдельный файл. Формат: `01_[theme].md`, `02_[theme].md`. Новая тема = новый файл. Дополнение темы = append в существующий файл.

**Файлы БЕЗ prepend (структурные, без изменений):**
- `architecture_map.md`
- `procedures.md`
- `knowledge_index.md`
- `specifications/`
- `checkpoint-latest.json`

**Как делать prepend:**
1. Прочитай файл (первые 20-30 строк)
2. Создай новую запись с датой/заголовком
3. Перезапиши файл: новая запись + старое содержимое

**Как работать с brainstorm/:**
1. При обсуждении новой темы: проверить `brainstorm/` — есть ли файл с такой темой?
2. Если НЕТ — создать `NN_[theme_name].md` (01_, 02_, 03_...)
3. Если ДА — добавить запись в конец существующего файла
4. Имена: английские буквы, цифры, подчёркивание (snake_case)

## 6. Правила автоматического обновления документации

### ⚠️ GIT COMMIT = ЕДИНСТВЕННЫЙ ТРИГГЕР (см. `documentation_sync_rules.md`)

**КАК ЭТО РАБОТАЕТ:**
1. Агент работает над задачами, НЕ обновляя документацию
2. По завершении каждой задачи: "Задача X выполнена. Готов продолжать."
3. Пользователь проверяет, даёт новые задачи
4. Пользователь командует: "GIT COMMIT"
5. Агент анализирует ВСЕ изменения сессии, обновляет документацию, делает commit

**Процедура GIT COMMIT:**
1. Проанализировать ВСЕ изменения сессии (код + файлы)
2. Определить файлы для обновления по карте триггеров
3. Обновить документацию (prepend + структурные файлы)
4. `git add . && git commit -m "[описание]"`
5. Сообщить: "Commit сделан. Документация обновлена: [список]"

## 7. Идеальный рабочий процесс с пользователем

```
1. Пользователь: "Хочу сделать X"

2. Оркестратор (в development mode):
   - "Я вижу несколько вариантов:
      Вариант А: [описание, плюсы, минусы, время]
      Вариант Б: [описание, плюсы, минусы, время]
      Есть риск: [описание риска]
     Что предпочитаете?"

3. Пользователь: "Вариант Б"

4. Оркестратор:
   - Добавляет задачу в tasks.md (APPEND — в конец файла)
   - Запускает architect субагента для планирования
   - Запускает code субагента для реализации
   - Сообщает: "Задача X выполнена. Готов продолжать."

5. Пользователь: "Сделай ещё Y"

6. Оркестратор:
   - Добавляет задачу в tasks.md
   - Запускает code субагента
   - Сообщает: "Задача Y выполнена. Готов продолжать."

7. Пользователь: "GIT COMMIT"

8. Оркестратор:
   - Анализирует ВСЕ изменения сессии (X + Y)
   - Обновляет документацию по карте триггеров
   - git add . && git commit -m "X и Y"
   - Сообщает: "Commit сделан. Документация обновлена."
```
