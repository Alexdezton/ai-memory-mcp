# DOCUMENTATION SYNC RULES — Детальный протокол обновлений

Этот стандарт описывает карту триггеров и алгоритм автоматического обновления документации по итогам сессии (команда `GIT COMMIT` или `EOS`).

---

## 1. Карта триггеров

### 1.1 Архитектура и структура
* **Новый компонент / модуль:** Обновить `architecture_map.md` (название, связи) и `knowledge_index.md` (маппинг файлов).
* **Удаление компонента:** Обновить `architecture_map.md` и `knowledge_index.md`, указав дату и причину удаления.

### 1.2 Данные и БД
* **Новая таблица / изменение схемы:** Обновить `specifications/DATA_MODEL.md` (схемы, связи, индексы).
* **API endpoint:** Обновить `specifications/API.md` (методы, пути, параметры, примеры ответов).

### 1.3 Бизнес-логика
* **Новая фича / изменение правил:** Обновить `specifications/[feature].md`. Если фичи нет, создать новую спецификацию.

### 1.4 Решения и задачи
* **Архитектурные решения:** Добавить новую запись в начало `other/decisions.md` (в формате ADR).
* **Задачи проекта:** Добавить или обновить статус в `tasks/tasks.md` (ACTIVE_TASK, NEXT_STEP).
* **Лог сессии:** Добавить резюме сессии в начало `session_log.md` (prepend).

### 1.5 Процессы
* **Команды запуска / сборки:** Обновить `procedures.md`.

---

## 2. Пошаговый процесс фиксации (GIT COMMIT / EOS)

1. **Анализ изменений:** Соберите список всех измененных и созданных файлов с последнего чекпоинта (`git diff` или `git log`).
2. **Синхронизация по карте триггеров:** Обновите соответствующие файлы документации в папке `AI_MEMORY/`.
3. **Запись лога:** Занесите 5-пунктовое резюме сессии в `session_log.md`.
4. **Коммит:** Выполните `git add .` и `git commit -m "[описание]"` (если запрошен коммит).
