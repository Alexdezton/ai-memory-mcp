# Протокол brainstorm/ — Правила работы с папкой совещаний

**⚠️ КРИТИЧЕСКОЕ ПРАВИЛО для агента:** `brainstorm/` — это НЕ prepend-файл. Это папка с отдельными файлами по темам.

## Формат именования файлов

Каждый файл — отдельная тема обсуждения. Нумерация по порядку создания:

```
01_[theme_name].md
02_[theme_name].md
03_[theme_name].md
...
```

**Правила именования:**
- Нумерация: `01`, `02`, `03`... (с ведущим нулём для сортировки)
- Имя темы: английские буквы, цифры, подчёркивание (snake_case)
- Пример: `01_kicked_blocked.md`, `02_cleanup_storage.md`
- **НЕ использовать:** пробелы, кириллицу в имени файла

## Создание файлов

**ПРИ ОБСУЖДЕНИИ НОВОЙ ТЕМЫ:**
1. Проверить `brainstorm/` — есть ли файл с такой темой?
2. Если НЕТ — создать новый файл: `[next_number]_[theme_name].md`
3. Если ДА — добавить запись в существующий файл (в конец)

**ПРИ ДОПОЛНЕНИИ СУЩЕСТВУЮЩЕЙ ТЕМЫ:**
1. Найти файл по теме
2. Добавить новую запись в конец файла (append)
3. Обновить TODO, если есть новые действия

## Структура файла

```markdown
# Owner: [Название темы]

**Дата:** [DD.MM.YYYY]

**ВАЖНО:** Это архив мозгового штурма. Актуальная информация о работе приложения находится в основных файлах AI_MEMORY. Содержимое этого файла могло устареть по ходу разработки.

## Проблема

[Описание проблемы или контекста]

## Обсуждение

**Вариант 1: [Название]**

[Описание варианта]

✅ Плюсы:
- ...

❌ Минусы:
- ...

**Вариант 2: [Название]** (ВЫБРАН)

...

## Принятое решение

[Краткое описание]

## Что изменится

[Конкретные изменения в БД, UI, функционале]

## Действия (TODO)

- [ ] Действие 2

## Технические детали (для агента)

[Код, SQL, технические детали для реализации]
```

## Когда использовать `brainstorm/`

- Обсуждение архитектурных решений
- Сравнение вариантов реализации
- Прототипирование идей
- Протоколы технических совещаний

## Когда НЕ использовать `brainstorm/`

- Реализация уже принятого решения → обновлять `specifications/`
- Мелкие багфиксы → обновлять session_log.md
- Принятые решения → decisions.md
