# DOCUMENTATION SYNC RULES — Протокол автоматического обновления

**Правило:** Документация AI_MEMORY обновляется ТОЛЬКО по команде `GIT COMMIT`. Агент работает, пользователь командует.

---

## 1. MANDATORY SYNC — Git Commit = Триггер обновления документации

### КАК ЭТО РАБОТАЕТ:

**ШАГ 1: Агент работает**
- Пишет код, меняет архитектуру, создаёт фичи
- НЕ обновляет документацию в процессе
- По завершении задачи сообщает: "Задача X выполнена. Готов продолжать."

**ШАГ 2: Пользователь проверяет**
- Смотрит результат, тестирует
- Может попросить исправить/доработать
- Может дать следующую задачу

**ШАГ 3: Пользователь командует: "GIT COMMIT"**
Это сигнал: всё готово, фиксируем.

**ШАГ 4: Агент делает commit-процедуру:**
1. Анализирует ВСЕ изменения текущей сессии (код + файлы)
2. Определяет какие файлы AI_MEMORY нужно обновить (по карте триггеров, Раздел 2)
3. Обновляет ВСЕ файлы документации
4. Делает `git add . && git commit -m "[описание]"`
5. Сообщает: "Git commit сделан. Обновлена документация: [список файлов]"

### ПОЧЕМУ ТАК:
- Агент мог работать над несколькими задачами подряд
- Документация должна соответствовать ВСЕМ изменениям сессии
- Пользователь контролирует когда фиксировать
- Git commit = точка невозврата. Документация = снимок этой точки.

**ЕСЛИ КОМАНДЫ "GIT COMMIT" НЕ БЫЛО = ДОКУМЕНТАЦИЯ НЕ ОБНОВЛЯЕТСЯ.**

## 2. КАРТА ТРИГГЕРОВ — Что изменилось → Какой файл обновить

### 2.1 Архитектура и структура

| Что изменилось | Файл | Тип обновления | Что записать |
|----------------|------|----------------|--------------|
| Новый компонент/модуль | `architecture_map.md` | Прямая запись | Имя, назначение, связи |
| Изменение связей между компонентами | `architecture_map.md` | Прямая запись | Что было → Что стало |
| Новый технологический стек | `project_overview.md` | Прямая запись | Технология, причина |
| Удаление компонента | `architecture_map.md` + `knowledge_index.md` | Прямая запись | Удалён, причина, дата |

### 2.2 Данные и БД

| Что изменилось | Файл | Тип обновления | Что записать |
|----------------|------|----------------|--------------|
| Новая таблица/коллекция | `specifications/DATA_MODEL.md` | Прямая запись | Схема, поля, индексы |
| Изменение схемы таблицы | `specifications/DATA_MODEL.md` | Прямая запись | Было → Стало, миграция |
| Новая связь между таблицами | `specifications/DATA_MODEL.md` | Прямая запись | Тип связи, поля |
| API endpoint | `specifications/API.md` | Прямая запись | Метод, путь, параметры, ответ |

### 2.3 Бизнес-логика и фичи

| Что изменилось | Файл | Тип обновления | Что записать |
|----------------|------|----------------|--------------|
| Новая фича | `knowledge_index.md` + `specifications/[feature].md` | Оба | Описание, логика, зависимости |
| Изменение бизнес-правила | `specifications/[feature].md` | Прямая запись | Что изменилось, почему |
| Новый алгоритм | `specifications/[feature].md` | Прямая запись | Описание, вход/выход, ограничения |
| Валидация/ограничения | `specifications/[feature].md` | Прямая запись | Правила, сообщения об ошибках |

### 2.4 Решения и совещания

| Что изменилось | Файл | Тип обновления | Что записать |
|----------------|------|----------------|--------------|
| Принято решение | `decisions.md` | **PREPEND** | Дата, решение, причина, файлы |
| Обсуждение на совещании | `brainstorm/[NN]_[topic].md` | Новый файл или APPEND | Формат: 01_theme.md, 02_theme.md. Новая тема = новый файл. Дополнение = в конец существующего |
| Архивация старого решения | `other/[DATE]_[name].md` | Новый файл | Архив по шаблону |

### 2.5 Задачи и сессии

| Что изменилось | Файл | Тип обновления | Что записать |
|----------------|------|----------------|--------------|
| Новая задача | `tasks.md` | **APPEND** | ID, описание, статус, приоритет |
| Задача выполнена | `tasks.md` | **APPEND** | ID → completed, результат |
| Задача заблокирована | `tasks.md` | **APPEND** | ID → blocked, причина |
| Начало сессии | `session_log.md` | **PREPEND** | Дата, цель, контекст |
| Конец сессии (commit) | `session_log.md` | **PREPEND** | Что сделано, NEXT_STEP |

### 2.6 Процедуры и процессы

| Что изменилось | Файл | Тип обновления | Что записать |
|----------------|------|----------------|--------------|
| Новый процесс/воркфлоу | `procedures.md` | Прямая запись | Название, шаги, триггеры |
| Изменение процесса | `procedures.md` | Прямая запись | Было → Стало |
| Новый тип субагента | `agent.md` | Прямая запись | Режим, назначение |

## 3. АЛГОРИМ GIT COMMIT — Пошаговый процесс

```yaml
# КОМАНДА ПОЛЬЗОВАТЕЛЯ: "GIT COMMIT" или "Выполни git commit"

# ШАГ 1: Анализ сессии
  Агент собирает ВСЕ изменения текущей сессии:
  - Какие файлы кода были созданы/изменены?
  - Какие файлы кода были удалены?
  - Были ли изменения архитектуры?
  - Были ли изменения БД/данных?
  - Были ли принятые решения?
  - Какие задачи выполнены/созданы?

# ШАГ 2: Определение файлов документации
  По карте триггеров (Раздел 2) агент определяет:
  - architecture_map.md — если менялась архитектура
  - specifications/ — если менялись данные/логика
  - decisions.md — если были решения (PREPEND)
  - tasks.md — если менялись задачи (PREPEND)
  - knowledge_index.md — если новые компоненты
  - session_log.md — конец сессии (PREPEND)

# ШАГ 3: Обновление файлов
  Для PREPEND файлов:
    1. Прочитать файл
    2. Создать запись с датой
    3. Перезаписать: новая запись + старое содержимое
  
  Для структурных файлов:
    1. Прочитать файл
    2. Обновить нужный раздел
    3. Записать обратно

# ШАГ 4: Git commit
  git add .
  git commit -m "[краткое описание всех изменений сессии]"

# ШАГ 5: Отчёт
  "Git commit сделан.
   Изменения в коде: [список файлов]
   Обновлена документация:
   - architecture_map.md — [что]
   - decisions.md — [что]
   - tasks.md — [что]"
```

## 4. ПРАВИЛА ДЛЯ СУБАГЕНТОВ

### 4.1 Code агент
- **НЕ ОБНОВЛЯЕТ** документацию
- Работает только с кодом
- По завершении сообщает что сделал

### 4.2 Architect агент
- **НЕ ОБНОВЛЯЕТ** документацию
- Работает только с планированием
- По завершении сообщает что предложил

### 4.3 Debug агент
- **НЕ ОБНОВЛЯЕТ** документацию
- Работает только с отладкой

### 4.4 Оркестратор
- **ЕДИНСТВЕННЫЙ** кто обновляет AI_MEMORY
- Делает это ТОЛЬКО по команде GIT COMMIT
- Перед commit проходит по карте триггеров

---

## 5. РАБОЧИЙ ПРОЦЕСС — Полный цикл

```
1. Пользователь: "Хочу сделать X"

2. Оркестратор (в development mode):
   - "Вижу варианты: А, Б. Что предпочитаете?"

3. Пользователь: "Вариант Б"

4. Оркестратор:
   - Обновляет tasks.md — добавляет задачу (PREPEND)
   - Запускает architect → plan
   - Запускает code → implementation
   - Сообщает: "Задача X выполнена. Готов продолжать."

5. Пользователь: "Сделай ещё Y"

6. Оркестратор:
   - Обновляет tasks.md — добавляет задачу
   - Запускает code → implementation
   - Сообщает: "Задача Y выполнена. Готов продолжать."

7. Пользователь: "GIT COMMIT"

8. Оркестратор (commit-процедура):
   - Анализирует ВСЕ изменения сессии (X + Y)
   - Обновляет architecture_map.md, decisions.md, specifications/
   - Обновляет session_log.md — конец сессии
   - Обновляет tasks.md — статусы задач
   - git add . && git commit -m "X и Y"
   - Сообщает: "Commit сделан. Документация обновлена."
```

---

## 6. EOS — End of Session

При команде EOS агент делает то же что и при GIT COMMIT:
1. Анализирует все изменения сессии
2. Обновляет документацию
3. Создаёт checkpoint-latest.json
4. Выводит: "CHECKPOINT SAVED: {ACTIVE_TASK, NEXT_STEP, BLOCKERS}"
