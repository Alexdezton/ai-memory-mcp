# DEVELOPMENT LOG — ИСТОРИЯ РАЗРАБОТКИ AI_MEMORY_MCP

Этот документ содержит полную историю проектирования, разработки, возникших проблем и архитектурных решений, принятых в ходе работы над MCP-сервером `ai-memory-mcp`.

---

## 1. Предыстория и цель проекта
Одной из основных проблем автономных агентов программирования (LLM) является тенденция писать огромные монолитные файлы (спагетти-код), игнорировать документацию и начинать писать код без четкого архитектурного плана. Это приводит к быстрому расходу токенов, забиванию контекстного окна и багам.

**Цель сервера:** Создать принудительные физические ограничения на уровне файловой системы, которые не позволяют агенту коммитить грязный код или отклоняться от дисциплины.
* Ограничение размера файлов кода (Hard limit: 1000 строк).
* Ограничение размера критического файла памяти `agent.md` (Hard limit: 600 строк). Для остальных файлов в `AI_MEMORY` строгие ограничения по длине сняты.
* Блокировка создания кода без наличия плана запуска (`project_launch_plan.md`).

## 2. Ключевые проблемы и их решения

### Проблема 1: Смещение рабочей директории (CWD)
* **Симптом:** При запуске MCP-сервера глобально через `npx` или локальный билд, среда Antigravity запускает процесс в своей собственной служебной папке (например, `C:\Users\...\AppData\Local\Programs\antigravity`). В результате папка `AI_MEMORY` инициализировалась внутри папки самой IDE, а не в проекте пользователя.
* **Решение:**
  - Мы добавили поддержку опционального аргумента `project_root` во все методы API (`init_project_memory`, `execute_loader`, `execute_eos`).
  - Была реализована функция `findProjectRoot(startPath)`, которая рекурсивно идет вверх по дереву папок от переданного пути или файла и ищет признаки папки `AI_Memory` или `.git`.

### Проблема 2: Потеря шаблонов при локальном запуске
* **Симптом:** При инициализации памяти (`init_project_memory`) сервер пытался скопировать шаблоны из `__dirname/../templates`. Однако при работе через npx из кэша npm или при отсутствии папки `templates` на диске сервер создавал пустую структуру.
* **Решение:**
  - В глобальный навык `SKILL.md` была добавлена **инструкция по резервному восстановлению**. Если сервер сообщает о предупреждении, агент обязан выполнить временное клонирование репозитория из GitHub (`git clone`), перенести шаблоны и немедленно выполнить `Remove-Item -Recurse -Force temp-ai-memory-mcp` для очистки диска.

### Проблема 3: Обход ограничений через нативные инструменты IDE
* **Симптом:** Агент обнаружил, что в IDE Antigravity ему доступен нативный инструмент `write_to_file`. Чтобы обойти ограничение `safe_write_file` в 1000 строк для CSS, он просто вызвал нативный метод, который записал файл без ограничений.
* **Решение:**
  - Мы внедрили функцию **сквозного контроля** `checkWorkspaceLimits()`.
  - Теперь при вызове **любого** MCP-инструмента (добавление лога, обновление задач, EOS) сервер сканирует проект. Если на диске обнаруживается файл, нарушающий лимиты, выполнение любого MCP-инструмента мгновенно блокируется с ошибкой `LIMIT_EXCEEDED`.
  - Также мы ввели строгий запрет в `core_rules.md` на искусственное сжатие/минимизацию кода. Единственным решением при превышении лимита строк стал рефакторинг на подключаемые модули.

### Проблема 4: Изменения файлов не отображались в UI Review Antigravity
* **Симптом:** Запись файлов напрямую через MCP-сервер (`safe_write_file` с использованием Node.js `fs.writeFileSync`) происходила в обход внутреннего трекера изменений IDE. В результате счетчик `Files Changed` оставался равен `0`, а измененные файлы не попадали в панель `Review` для проверки пользователем.
* **Решение (Переход от safe_write_file к verify_write):**
  - Мы удалили инструмент `safe_write_file` из схемы сервера.
  - Вместо него был добавлен инструмент-валидатор **`verify_write(filepath)`**.
  - **Новый протокол:** Агент записывает код с помощью встроенного инструмента IDE `write_to_file` (что гарантирует 100% отображение изменений в Review UI), а затем сразу вызывает `verify_write(filepath)`.
  - Инструмент `verify_write` считывает записанный файл с диска, проверяет его размер, структуру, правила пути и наличие плана запуска. Если требования нарушены — выдается ошибка, блокирующая сессию.
  - **Нулевой перерасход токенов:** Поскольку в `verify_write` передается только путь к файлу, а не его содержимое, мы полностью избежали двойной отправки текста кода через сеть (экономия до 90% токенов на записи файлов).

### Проблема 5: Чрезмерные ограничения на файлы документации в AI_MEMORY
* **Симптом:** В процессе работы над документацией или логами файлы внутри `AI_MEMORY` могли естественным образом расти, упираясь в жесткий лимит 600 строк. Однако ограничения важны только для файла `agent.md` (чтобы экономить контекст модели), в то время как подробные спецификации или процедуры должны быть полноценными.
* **Решение:**
  - Мы сняли проверку лимита строк со всех файлов внутри папки `AI_MEMORY`, **кроме файла `agent.md`** (или `AGENT.md`), для которого сохраняется лимит `max_memory_md_lines` (по умолчанию 600 строк). Все остальные документы памяти теперь могут иметь произвольный объем.

### Проблема 6: Необходимость автоматического логирования работы с MCP
* **Симптом:** Пользователю было сложно отслеживать, когда именно агент обращался к серверу MCP, с какими параметрами и какие ошибки возникали в процессе проверки.
* **Решение:**
  - Мы внедрили функцию **автоматического логирования всех вызовов MCP** (`logMcpAction`).
  - Все запросы, аргументы, успешные результаты и ошибки валидации автоматически записываются сервером в файл [`AI_MEMORY/mcp_history.log`](file:///d:/ai/project/test-ia-memory-mcp/AI_MEMORY/mcp_history.log).
  - Запись производится скрытно и асинхронно на уровне Node.js-сервера, что гарантирует **нулевой расход токенов** LLM и полное отсутствие влияния на производительность.
